## 一、项目定位与核心价值

### 1.1 一句话定义

这是一个 **Tauri 2 桌面应用**，通过"样本文本驱动"的 5 步结构化流程，将汽车维修手册中的非结构化文本自动转化为结构化数据（JSON/CSV/Excel/Markdown）。

### 1.2 解决的实际问题

汽车维修手册包含大量非结构化信息（故障症状表、诊断流程、DTC 故障码、拆装步骤、扭矩规格等），传统方式依赖人工逐条阅读和手工录入，存在以下痛点：

| 痛点 | 本项目的解决方案 |
|------|-----------------|
| 人工录入效率低、易出错 | AI 自动生成 Python 抽取脚本，批量处理 |
| 不同手册格式差异大 | 14 个场景模板覆盖主流格式，支持 5 种症状子表型 |
| 需要专业领域知识 | LLM 辅助分析 + 本地规则引擎双通道 |
| 无网络/API 时无法工作 | 完善的三级降级策略（LLM → 本地规则 → 硬编码默认值） |
| 抽取规则难以复用 | 规则分析结果可持久化、会话可恢复 |

### 1.3 业务需求映射

```
用户需求                        对应功能
─────────────────────────────────────────────────────
"我要从这段文本中提取数据"    → Step1: 获取样本文本
"这段文本属于什么类型"        → Step2: AI/本地双通道场景识别
"按什么规则提取"              → Step3: LLM 规则分析 + 提示词工程
"生成可执行的抽取脚本"        → Step4: AI 脚本生成（流式展示）
"批量处理并导出结果"          → Step5: Python 脚本执行 + 多格式输出
```

## 三、核心模块分析

### 3.1 五步状态机

**设计亮点**：四值枚举 `pending → current → done → stale`，配合严格的前置校验和级联重置。

```
Step1 done → Step2 current → Step2 done → Step3 current → ...
                                    ↑
                            修改 Step2 输入时
                            Step3-5 全部降级为 stale
```

- `canEnterStep(N)`：要求 Step 1 到 N-1 全部为 `done`
- `clearDownstreamFromStep(N)`：清除 N 之后所有步骤的数据和状态
- 路由守卫 `router.beforeEach` 强制执行顺序

### 3.2 场景模板系统（14 个场景）

**9 个主场景 + 5 个子场景**，覆盖汽车维修手册的核心数据类型：

| 优先级 | 场景 | 业务含义 |
|--------|------|----------|
| P0 | symptom_table | 故障症状表（5 种子表型） |
| P0 | check_confirm_text | 检查与确认流程 |
| P0 | warning_notice | 安全警告/注意事项 |
| P0 | repair_plan_steps | 维修方案（容器，含 5 子场景） |
| P1 | diagnostic_flow | 诊断流程表（分支/跳转） |
| P1 | visual_inspection_table | 外观检查表 |
| P1 | specification_table | 规格参数表 |
| P1 | torque_spec | 扭矩规格表 |
| P2 | dtc_extraction | DTC 故障码表 |

**子场景**（均属 repair_plan_steps）：repair_remove、repair_install、repair_tool、repair_cleaning、repair_inspection

### 3.3 场景路由算法

**双通道识别**：

1. **AI 通道**：`classify_scene_ai` → LLM 分类（需 API Key）
2. **本地通道**：`sceneRouter.ts` → 加权评分算法（无网络可用）

评分规则：
- 关键词命中：+2/项
- 表头别名命中：+3/项
- 结构模式命中：+8/项
- 场景特化加分（如 DTC 码正则 +14，症状三元组 +16）
- 负向惩罚（如症状表中出现流程词 -16）

### 3.4 LLM 集成架构

**核心特性**：所有 LLM 功能都有本地降级，确保离线可用。

| 功能 | LLM 方式 | 本地降级 |
|------|---------|---------|
| 场景分类 | GPT 分类 | 关键词评分排序 |
| 规则分析 | LLM 结构化输出 | 模板解析 + 置信度评分 |
| Prompt 优化 | LLM 文本生成 | 硬编码预设模板 |
| 别名映射 | LLM 生成同义词 | 模板 header_alias 合并 |
| 脚本生成 | LLM 代码生成 | scaffold 模板渲染 |

**关于"流式"**：当前的流式实现是**伪流式** — 先完整获取 LLM 响应，再按 180/240 字符分块 + 12ms 间隔模拟逐字显示效果。非真正的 SSE 流式传输。

### 3.5 Python 执行流程

```
Rust (run_generated_script)
  ├── 写入 extract-{run_id}.py（scaffold + 场景适配 JSON）
  ├── 写入 config-{run_id}.json
  ├── spawn python runner.py --script ... --config ...
  ├── 实时读取 stdout/stderr → emit("python-log") → 前端
  └── 解析 OUTPUT_FILE:: 标记获取输出路径
```

抽取策略优先级：表格解析（`|` 分隔）→ 键值对（正则匹配）→ 编号步骤 → 全文截断 500 字。

### 5.2 症状场景的特殊设计

`symptom_table` 是全系统最复杂的场景，支持 5 种子表型：
1. 标准三列表（症状 + 原因 + 措施）
2. 故障现象三列表
3. 续行三列表（当前行症状为空，继承上一行）
4. 现象-原因二列表（降置信度）
5. 标题症状的二列表（需标题存在唯一明确症状）

路由引擎中实现了 +16 分的正向三元组检测和 -16 分的负向排除（流程词/DTC 词），形成对称的歧义消解机制。

## 八、总结

本项目在**业务设计层面**表现出色：5 步状态机严谨、14 个场景模板覆盖全面、三级降级策略确保离线可用、症状场景的 5 种子表型设计精巧。

在**工程实现层面**存在改进空间：Rust 单文件架构需拆分、前端 Store 职责过多、代码重复率较高、测试覆盖几乎为零、安全防护需要加固。

**核心价值**：将 AI 能力（LLM 规则分析 + 脚本生成）与确定性执行（Python 脚本运行）分离，既利用了 AI 的理解能力，又保证了运行阶段的稳定性和可复现性。这是一个务实的架构选择。

**最大风险**：Python 脚本执行缺乏沙箱隔离，LLM 返回内容未经充分验证。在投入生产前应优先解决这两个安全问题。
